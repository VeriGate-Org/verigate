Deeds RFP Requirement Status Matrix

This matrix reflects the current codebase after the provider-independent deeds work already implemented.

Status legend:

  • Covered: implemented in a materially usable way in code.
  • Partial: implemented in part, but still missing important scope or production hardening.
  • Scaffold: implemented as a placeholder or heuristic shell, not yet RFP-grade.
  • Blocked: not finishable without the external DeedsWeb/provider contract or external datasets.
  • Operational: requirement is outside product code and still needs business/ops delivery proof.

Overall Position

What is materially covered in code now

  • Web portal access and navigation for deeds search, reports, settings, map, conversion, and valuation.
  • Cached deeds reports, exports, watches, alerts, audit events, and delegated deeds team management.
  • Provider-independent map, conversion, and valuation scaffolding.

What is still the main blocker

  • Real provider-backed deeds/document/DOTS behavior remains blocked by the external DeedsWeb contract and related datasets.

What still falls short even without the provider

  • Spatial and conversion accuracy are not RFP-grade yet.
  • Valuation is placeholder logic only.
  • Scheduling/integration/compliance/support/onboarding requirements are not fully closed by code.
